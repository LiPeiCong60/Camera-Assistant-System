# Camera Assistant 项目总览与架构说明

本文档提供当前项目的总览版说明；完整运行机制、模块内部职责和时序图见 [技术架构与运行机制详解](./技术架构与运行机制详解.md)。

## 1. 项目目标

Camera Assistant 面向手机拍摄辅助和设备联动场景，目标是把手机拍摄、模板构图、AI 图片分析、本地云台设备和运营后台组织成一个可联调、可演示、可继续扩展的系统。

当前能力包括：

- 手机端登录、注册、拍摄、模板叠加、历史记录、AI 分析和设备联动。
- 设备端本地会话、实时视频处理、人体检测、模板构图、云台控制、抓拍、AI 背景锁和预览。
- 手机与设备端的主视频链路已改为 WebRTC。
- WebSocket/JPEG 旧链路仍保留为 fallback。
- 后端提供账号、套餐、模板、抓拍记录、AI 任务和上传文件管理。
- 管理后台提供用户、套餐、设备、推荐模板、AI Provider、抓拍和任务管理。

## 2. 总体架构

```mermaid
flowchart LR
  Mobile["mobile_client\nFlutter App"] -->|/api/mobile/*| Backend["backend\nFastAPI"]
  Admin["admin_web\nVue"] -->|/api/admin/*| Backend
  Backend --> DB["PostgreSQL"]
  Backend --> Uploads["uploads 静态目录"]
  Mobile -->|/api/device/* 控制| Runtime["device_runtime\nFastAPI"]
  Mobile <-->|WebRTC video + preview\nPOST /api/device/webrtc/offer| Runtime
  Mobile -. fallback .->|WS /stream/mobile-ws| Runtime
  Runtime -. fallback .->|WS /preview-ws\npreview.jpg| Mobile
  Runtime --> Gimbal["TTL bus/mock 云台"]
  Runtime --> Captures["设备本地 captures"]
```

业务数据走 `backend`，实时设备控制和视频走 `device_runtime`。这两个服务地址在手机设置页中分别配置为“业务后台地址”和“设备运行时地址”。

## 4. 核心流程

### 4.1 手机独立拍摄

1. 手机登录 `backend`。
2. 手机加载套餐、模板和历史。
3. 拍摄页打开本机摄像头并绘制模板/姿态 overlay。
4. 拍照后上传到 `POST /api/mobile/captures/file`。
5. 创建抓拍记录或 AI 任务。
6. 历史页从 `backend` 读取记录。

### 4.2 设备联动

1. 手机设置页配置设备运行时地址，例如 `http://192.168.1.30:8001`。
2. 设备联动页打开 `mobile_push` session。
3. 手机通过 `flutter_webrtc` 获取摄像头 video track。
4. 手机向 `POST /api/device/webrtc/offer` 发送 SDP offer。
5. 设备端 `aiortc` 返回 answer。
6. 手机摄像头画面进入 `mobile_push_frame_store`。
7. 设备端帧循环执行检测、构图、云台控制和 overlay 渲染。
8. 设备端通过 WebRTC preview video track 返回处理后画面。
9. WebRTC 失败时自动落到 WebSocket/JPEG fallback。

### 4.3 模板引导

模板可来自业务后端用户模板或设备本地模板。设备端在 `SMART_COMPOSE` 模式下把当前稳定人体框与模板目标进行比较，输出构图反馈、ready 状态和可选自动控制目标点。

### 4.4 AI 分析

业务后端 AI 负责手机照片分析、背景分析和连拍选优，Provider 由管理后台配置。设备端 AI 负责编排本地自动找角度、背景锁和锁定框匹配，它直接操作设备帧和云台。

## 5. 当前关键结论

- 设备联动已经是 WebRTC 主链路，不再只是 WebSocket 拆帧。
- fallback 没有删除，可继续用于调试和兼容。
- 真机不能使用 `127.0.0.1` 或 `10.0.2.2`，必须使用局域网 IP。
- Android 已声明摄像头、网络、WebRTC 相关权限和 cleartext。
- 前摄镜像和 overlay 对齐逻辑属于画面坐标/渲染层，WebRTC 改造不应破坏该逻辑。
- 设备运行时是单会话模型，打开新 session 会关闭旧 session。
- 设备抓拍和业务后端抓拍目前是两个存储域。
