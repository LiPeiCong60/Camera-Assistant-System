## 1. 执行摘要

`Camera Assistant` 是一个围绕“手机拍摄辅助 + 设备联动 + 云端 AI 分析 + 后台管理”展开的多端系统。当前主仓库已经不是简单原型，而是一个具备真实主链路的组合项目：

- `backend` 负责业务中心、账号鉴权、拍摄记录、AI 任务和后台管理接口
- `device_runtime` 负责本地实时视觉循环、云台控制、模板引导和设备 API
- `mobile_client` 负责用户主入口、拍照、模板、上传分析、历史记录和设备联动
- `admin_web` 负责管理端登录、用户/套餐/设备/抓拍/AI 配置管理
- `demo` 保留了更早期的原型与大量复用逻辑，是当前设备运行时的重要来源

当前项目最大的价值不在某一个单点功能，而在于“跨端闭环”已经基本成型：手机端可以发起拍摄与 AI 分析，后端可以落库并调 AI，设备端可以接收控制和 AI 建议，后台可以查看管理数据与配置 Provider。

同时，项目也存在明确的阶段性边界：

- 更偏“可运行、可演示、可继续迭代”的工程版本
- 不是已经收口到线上生产标准的正式商业系统
- 真正欠缺的主要不是页面数量，而是一些关键闭环和工程化收尾

## 2. 项目定位与目标

根据 `docs/开发定稿.md`、`docs/项目当前状态说明.md` 与代码现状综合判断，项目目标可以概括为：

1. 提供一个手机端主导的拍摄辅助入口
2. 在有树莓派云台设备时，支持局域网联动、模板引导、实时跟踪和 AI 锁机位
3. 将单图分析、背景分析、连拍选优等非强实时能力交给云端 AI
4. 用后台统一管理用户、套餐、设备、拍摄记录、AI Provider 和推荐模板

项目支持两种使用模式：

1. 手机独立模式
   - 手机本地拍照、模板、叠加层、上传 AI
2. 树莓派增强模式
   - 手机仍是主入口，树莓派负责实时控制闭环

这意味着项目不是单体 App，也不是单纯设备固件，而是一个多端协作系统。

## 4. 总体架构与主链路

### 4.1 四层架构

系统可以抽象为四层：

1. 交互层：`mobile_client`
2. 设备执行层：`device_runtime`
3. 业务中台层：`backend`
4. 运维管理层：`admin_web`

辅助层：

- `database/schema.sql`
- `docs/*`
- `demo/*`

### 4.2 主链路 1：手机拍照并上传 AI 分析

1. 用户在手机端登录
2. 手机端创建拍摄会话
3. 手机端拍照
4. 手机端把文件上传到 backend `/mobile/captures/file`
5. backend 落盘到 `uploads/...`
6. 手机端再调用 `/mobile/captures/upload` 写抓拍记录
7. 手机端调用 `/mobile/ai/analyze-photo` 或 `/mobile/ai/analyze-background`
8. backend 按套餐/默认配置选择 AI Provider
9. backend 调外部多模态模型
10. backend 记录 `ai_tasks`
11. 手机端显示 AI 摘要、评分、建议框或角度建议

### 4.3 主链路 2：连拍选优

1. 手机端在 AI 连拍模式下采集多张图片
2. 每张图先上传文件，再写抓拍记录
3. 手机端调用 `/mobile/ai/batch-pick`
4. backend 将多个抓拍图发送给 AI Provider
5. backend 返回最佳图片 ID，并将对应 `captures.is_ai_selected=true`
6. 手机端历史页可展示“最佳图”结果

### 4.4 主链路 3：设备联动

1. 手机端输入设备地址或使用最近连接配置
2. 手机端调用 `device_runtime` 健康检查
3. 手机端打开设备会话并传入 `session_code`、`stream_url`
4. `device_runtime` 启动本地实时循环、视频读取、检测、跟踪和预览
5. 手机端可以进行：
   - 手动控制
   - 切换模式
   - 切换跟随模式
   - 下发模板
   - 触发抓拍
   - 将 backend 返回的 AI 锁机位/角度建议下发给设备执行

### 4.5 主链路 4：后台管理

1. 管理员登录后台
2. 查看统计、用户、套餐、设备、抓拍、AI 任务
3. 维护 AI Provider 配置
4. 维护推荐默认模板
5. 通过套餐 `feature_flags` 控制用户使用的 AI 配置优先级

### 5.5 数据库初始化方式

当前数据库初始化方式不是标准迁移框架，而是：

1. `Base.metadata.create_all(engine)`
2. 手动执行一组兼容性 `ALTER TABLE` 语句

优点：

- 启动门槛低
- 适合项目早期快速迭代

局限：

- 缺少 Alembic 这类标准迁移历史
- 后续多人协作和线上版本演进时，数据库变更可追踪性不足

### 5.6 鉴权实现

backend 使用了自定义 HMAC Token，而不是标准 JWT 库。

优点：

- 依赖少
- 可控

当前实现特征：

- `hash_password` 使用 `pbkdf2_hmac("sha256")`
- access token 包含：
  - `user_code`
  - `role`
  - `iat`
  - `exp`
- 通过 `Authorization: Bearer <token>` 解析

需要注意：

- 默认密钥 `camera-assistant-dev-secret` 只适合开发环境
- 没有 refresh token 体系
- 不是行业通用 JWT 方案，后续接第三方鉴权或网关时需额外适配

### 5.8 MobileService 业务逻辑

`MobileService` 是手机端业务主中心，当前职责包括：

- 套餐与订阅读取
- 模板创建与删除
- 拍摄会话创建
- 抓拍记录写入
- 三类 AI 任务创建：
  - `analyze_photo`
  - `analyze_background`
  - `batch_pick`

关键特征：

### 5.9 AI Provider 选择与调用

这是 backend 最有工程价值的部分之一。

当前设计：

1. 先看用户当前订阅套餐
2. 从套餐 `feature_flags` 里读取：
   - `default_ai_provider_code`
   - `available_ai_provider_codes`
3. 如果没有，再回退到系统默认 Provider

Provider 当前重点支持：

- `provider_format = openai_compatible`
- `vendor_name = longcat` 做了专门兼容逻辑

`AiProviderService` 已实现：

- 图片转 data URL / base64
- 结构化 prompt
- 多图批量选优
- LongCat 多种 payload 兼容重试
- 非结构化 JSON 的兜底解析
- 日志脱敏与错误摘要

这是当前项目里“工程含量最高”的模块之一，说明系统已经不只是 mock 接口。

### 5.10 上传与模板自动生成

backend 目前已实现两条很关键的产品化链路：

1. 图片上传到 backend 本地静态目录

`TemplatePoseService` 直接复用了 `device_runtime` 中的模板生成与视觉检测能力，这种跨目录复用让“手机模板创建”与“设备模板引导”共享了一套姿态语义。

- 自动修复可疑模板
- 基于人体姿态构造模板 profile
- 运行时评估当前画面与模板的匹配度
- 返回：
  - `pose_score`
  - `compose_score`
  - `total_score`
  - `messages`
  - `target_norm`

这让设备端不只是“执行控制”，而是真正具备模板引导能力。

### 6.8 FrameProcessor

`FrameProcessor` 统一了实时处理 Pipeline：

1. 选择稳定目标
2. 可靠检测过滤
3. 更新 `RuntimeState`
4. SMART_COMPOSE 模式下做模板评估
5. 计算是否需要自动跟随
6. 输出跟踪命令

这是设备端最重要的实时业务中枢。

### 6.9 设备抓拍与预览

当前设备端抓拍和预览策略都比较务实：

- 抓拍：直接保存到本地 `captures/<date>/...jpg`
- 预览：通过 `/api/device/preview.jpg` 返回“最新一帧 JPEG”

优点：

- 容易实现
- 演示友好
- 对移动端接入简单

局限：

- 不是低延迟视频播放方案
- 不是完整的设备抓拍回传 backend 闭环

### 6.10 AI 编排代码的现状

`services/ai_orchestrator.py` 和 `interfaces/ai_assistant.py` 里仍保留了较完整的本地 AI 批量扫描、角度搜索、背景扫描设计，但当前设备 API 主链并没有把这些能力串进 `DeviceSessionContext`。

也就是说：

- “设备端自主发起 AI 搜索”这套能力存在代码
- 但“当前正式主链”更偏向：
  - backend 做 AI
  - 手机取结果
  - 设备只执行 `apply-angle` / `apply-lock`

这是一种很重要的阶段性取舍。

### 6.11 device_runtime 优势

- 实时循环清晰
- 控制、检测、模板、状态职责分离较好
- mock 与真实 TTL 总线舵机切换机制明确
- 预览、状态、控制接口都具备
- 已达到“设备联动可演示”的水平

### 6.12 device_runtime 风险与问题

1. `device_runtime/main.py` 仍是占位入口，容易误导新接手人员
2. `ControlService.manual_move()` 与 `app_core.process_command()` 的上下方向定义不一致
3. 设备抓拍没有自动回流 backend，导致“设备现场最新图”和“backend AI 最新图”可能不是同一张
4. 预览仍是 JPEG 轮询，不是实时视频
5. 真实方向、home 角、舵机正负方向仍需要真机校准

## 13. 风险、债务与审查意见

以下内容是本次审查中最值得关注的部分。

### 13.1 架构层风险

1. 当前存在“正式模块 + demo 原型”双来源并行结构
   - 优点是复用快
   - 风险是后续维护时容易出现“正式代码在这边，测试在那边，思路在另一边”

2. backend、device_runtime、mobile 三端已经形成较强耦合
   - 当前是优势
   - 后续如果接口频繁改动，联调成本会快速上升

### 13.2 实现层风险

1. `device_runtime/main.py` 与真实启动方式不一致
2. 云台上下方向语义存在代码层不一致
3. 设备抓拍与 backend AI 记录还未完全闭环
4. 设备联动依赖手工输入地址，不利于产品完整性
5. 移动端关键页面体量过大，后续重构压力高

### 13.3 测试层风险

自动化测试分布不均衡：

- `demo/tests` 有一组 Python 测试
- `mobile_client/test` 有少量 Flutter 测试
- backend 没有看到独立测试目录
- device_runtime 正式模块没有看到独立测试目录
- admin_web 未见测试代码

这意味着当前测试更多验证“原型核心算法”和“移动端局部页面/错误文案”，但对正式多端主链缺少系统性覆盖。

### 13.4 安全与运维层风险

1. backend / device_runtime 都是开放 CORS
2. backend 存在默认开发密钥
3. 本地磁盘上传方案不适合生产多实例
4. 缺少标准迁移工具
5. 缺少统一日志、监控、审计设计

## 14. 建议的下一阶段优先级

### P1：必须先补的闭环

1. 打通“backend 设备绑定 -> mobile 设备选择”
2. 打通“device_runtime 抓拍 -> backend Capture -> backend AI -> 再回设备”
3. 完成真机云台方向与 home 角校准

### P2：工程化收口

1. backend 引入数据库迁移体系
2. backend / device_runtime / admin_web 增加正式测试
3. 拆分移动端大页面
4. 统一错误码与日志结构

### P3：产品体验升级

1. 设备预览升级为更实时的视频方案
2. 后台增加更细的统计与筛选
3. 手机端做更稳定的设备连接记忆与设备列表化

## 15. 最终结论

这是一个已经跨过“空壳原型期”的多端项目。

从代码结构、模块数量、接口深度和文档质量来看，项目当前最适合的定位是：

- 一个可运行、可讲解、可演示、可继续迭代的智能拍摄系统工程版本

它当前最值得肯定的地方，是四端都已经有真实代码和真实角色，而不是只把设想写进文档。

它当前最需要补的，不是再加很多新页面，而是补齐几条关键闭环和工程化收尾：

- 设备绑定闭环
- 设备抓拍回流闭环
- 真机控制校准
- 测试、迁移、安全与部署收口

如果这几个点补齐，项目就会从“能演示的完整系统”进一步走向“可稳定交付的系统版本”。
