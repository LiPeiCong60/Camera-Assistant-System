# Camera Assistant Codex 执行开发计划

## 1. 文档目的

本文档不是产品愿景说明，而是后续交给 Codex 按阶段落地的执行计划。

目标：
- 明确开发顺序
- 明确每阶段交付物
- 明确每项任务的输入、输出、依赖与验收标准
- 降低多人并行开发时的边界不清问题

本文档与 `docs/开发定稿.md` 配套使用：
- `开发定稿.md` 负责定义方向与最终方案
- 本文档负责定义 Codex 该如何一步一步实现

## 2. 执行总原则

### 2.1 核心原则

1. 优先跑通主链路，不优先追求完美架构
- 先让系统可运行、可演示、可联调
- 再逐步优化抽象、命名、解耦和工程化

2. 先实现 60 分版本，再扩展高级功能
- 优先做手机独立模式
- 再做树莓派联动
- 再做云端 AI 闭环
- 最后补管理后台

3. 强复用 `demo` 的有效资产
- 不重写已经证明可用的检测、跟踪、云台、模板逻辑
- 优先迁移和整理，而不是推翻重做

4. 明确分层，避免职责混乱
- 手机端负责交互、拍照、上传、转发 AI 结果
- 树莓派负责实时视觉与设备控制
- 后端负责业务数据与云端 AI 中转
- 管理端负责后台管理

5. 每一阶段都必须有验收标准
- 没有验收标准的任务不算完成

### 2.2 Codex 执行规则

后续使用 Codex 时，默认按以下规则执行：

1. 每次只聚焦一个明确阶段或一个明确子任务
2. 优先先补目录、骨架、契约和最小闭环
3. 尽量不要在同一轮同时大改手机端、树莓派端、后台端
4. 每完成一阶段必须补文档或注释，避免后续失忆
5. 任何从 `demo` 迁移的模块，都要先以“最小改动迁移”为主
6. 没有必要时，不做大规模重命名
7. 未跑通前，不做过度抽象

## 3. 完成定义

### 3.1 第一版完成定义

第一版完成时，应满足：

- 手机端可以独立打开摄像头并显示基础叠加层
- 手机端可以导入模板并显示模板线
- 手机端可以拍照并上传后端
- 树莓派可以接收视频流并控制云台
- 树莓派可以执行人物追踪
- 树莓派可以执行模板引导自动调机位
- 后端可以接入云端 AI 并返回结构化 JSON
- 手机端可以把 AI JSON 转发给树莓派
- 树莓派可以根据 JSON 执行锁机位
- 管理端可以查看用户、套餐、拍摄记录和 AI 任务

### 3.2 不属于第一版完成定义

以下内容不计入第一版必须完成项：

- 完整支付系统
- 复杂权限系统
- 多租户
- 本地大模型部署
- 高级运营系统
- 多设备并行协同

## 4. 推荐开发顺序

严格建议按以下顺序执行：

1. 仓库初始化与目标目录骨架
2. 树莓派运行时 `device_runtime`
3. 业务后端 `backend`
4. 手机端 `mobile_client`
5. 树莓派联动与 AI 闭环联调
6. 管理端 `admin_web`
7. 联合测试、性能调优、文档补齐

这样安排的原因：
- `device_runtime` 复用 `demo` 最多，最容易尽快形成核心能力
- `backend` 需要尽早建立数据模型与业务接口
- `mobile_client` 要建立在稳定接口之上
- `admin_web` 与主拍摄链路相比优先级更低

- 每个目录有模块职责说明

验收标准：
- 项目结构与 `开发定稿.md` 保持一致
- 每个主目录不再是空目录

### 任务 M0-2 落地共享契约文档

目标：
- 避免三端接口理解不一致

Codex 任务：
- 新建接口契约文档
- 明确以下对象结构：
  - 用户
  - 模板
  - 拍摄会话
  - 抓拍记录
  - AI 任务
  - AI 锁机位 JSON

- `docs/接口契约.md`

目标：

  - `templates`

  - `captures`
  - `ai_tasks`

- `database/schema.sql`

验收标准：
- 表结构能支撑第一版接口需求
- 主键、时间字段、状态字段齐全

建议：

## 7.4 管理端推荐实现

建议：
- Vue 3

- Element Plus

## 8. Codex 使用建议

后续给 Codex 下任务时，建议按以下粒度发起，不要一口气让它做完整项目。

### 推荐任务粒度

好的任务粒度示例：
- “初始化 `backend` 的 FastAPI 骨架并创建健康检查接口”
- “把 `demo` 的检测和跟踪模块迁移到 `device_runtime` 并修复导入”
- “创建 Flutter 手机端摄像头页和拍照页骨架”
- “为后台创建用户列表和套餐列表页面”

不推荐的任务粒度示例：
- “把整个项目全部开发完”
- “把三端一起写完”

### 推荐执行策略

1. 先让 Codex 做骨架
2. 再让 Codex 做单模块最小闭环
3. 再让 Codex 做接口打通
4. 最后让 Codex 做联调与修补

## 9. 推荐的 Codex 执行批次

建议实际执行时按以下批次推进：

### 批次 1
- 建立 `backend`
- 建立 `device_runtime`
- 建立 `database/schema.sql`
- 建立 `docs/接口契约.md`

### 批次 2
- 迁移 `demo` 核心模块到 `device_runtime`
- 跑通树莓派手动控制与检测状态接口

### 批次 3
- 建立 `backend` 数据模型与基础业务接口

### 批次 4
- 建立 Flutter 手机端骨架
- 跑通摄像头、拍照、基础叠加

### 批次 5
- 手机端接树莓派接口
- 打通手动控制

### 批次 6
- 接入云端 AI
- 打通 AI JSON 到树莓派执行

### 批次 7
- 建立管理后台

### 批次 8
- 全链路联调

## 10. 当前建议的第一批立刻执行任务

如果立刻开始，建议 Codex 先做下面 4 项：

1. 创建 `backend`、`device_runtime`、`admin_web`、`database` 的工程骨架
2. 生成 `database/schema.sql`
3. 生成 `docs/接口契约.md`
4. 把 `demo` 的核心实时控制模块迁移到 `device_runtime`

原因：
- 这 4 项完成后，整个项目才真正进入可开发状态
- 后续手机端、后台端都能基于稳定边界并行开发

## 11. 结论

本项目的 Codex 执行顺序必须坚持：

`先骨架 -> 再树莓派实时内核 -> 再业务后端 -> 再手机端 -> 再 AI 闭环 -> 再管理端 -> 最后联调`

不建议一开始就同时推进所有端。

本文档将作为后续 Codex 分阶段执行、任务拆分与验收的直接依据。
